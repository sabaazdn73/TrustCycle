01 Problem Statement

Academic admissions have largely modernised. Application portals at leading European universities — including Warwick, University College Dublin, and institutions across the EU — are digital, structured, and efficient. Formal academic credentials are increasingly covered by the European Digital Credential (EDC) framework, providing verifiable, tamper-proof records of degrees, transcripts, and certifications. This part of the academic chain is, for the most part, solved.

One gap remains: the recommendation letter.

Despite the maturity of surrounding infrastructure, top-ranked European universities still handle recommendation letters through fragile, email-based workflows. A typical admission portal asks for the professor's full name, official email, and phone number — then sends an automated message requesting a PDF upload or institutional email submission. The student has no visibility. The university has no cryptographic guarantee of authenticity. The professor has no unified interface.

Consider a PhD applicant targeting 20 universities across Europe, requiring two referees per programme. That is 80 separate recommendation requests, tracked manually, with no reliable confirmation layer. Talented candidates are rejected not because of their qualifications, but because an email was lost or a deadline was missed silently.

"University recommendation letters are the last unverified, non-portable document in an otherwise modernising academic admission chain."

02 Why This Problem Fits IOTA

The missing ingredient is not a new platform. Universities do not need to replace their existing portals. What is missing is a trust layer — a mechanism that cryptographically binds a professor's authority to a credential, anchors it immutably, and lets any verifier confirm it with a single reference key.

This is precisely what the IOTA Trust Framework provides:

IOTA ServiceWhy ChosenWhat It Does in TrustCycle
IdentityBridge real-world academic standing to digital authorityGrants IssuerAuthorization capability — only verified professors can mint credentials
NotarizationOnce issued, a recommendation must be immutable and timestampedContent hash + passport hash anchored as Move Shared Object on IOTA L1

TrustCycle does not ask universities to change their platforms. It asks them to accept one additional field — a reference key — the same way they accept an ORCID identifier today. Behind that key is cryptographic proof anchored permanently on the IOTA ledger.

03 Solution Overview

TrustCycle replaces the email-based recommendation loop with a "Write Once, Verify Anywhere" architecture. The core idea: a professor writes one recommendation, it is anchored on-chain, and the student carries a portable reference key that any university can verify instantly — no emails, no uploads, no intermediaries.

05 Smart Contract Design

The Move contract models academic trust using capability-based governance — not NFTs or token transfers. Recommendations are shared on-chain domain aggregates: publicly queryable, issuer-revocable, content-immutable.

ObjectTypePurpose
RecommendationMove Shared ObjectCore credential. Stores only hashes. Content is immutable; only active state is mutable.
IssuerAuthorizationCapability ObjectHeld by each accredited professor. Only active holders can issue or revoke.
ProtocolConfigMove Shared ObjectGovernance object. Controls issuance and revocation system-wide.
UniversityCapCapability ObjectMaster authority for accrediting professors.

Domain invariants enforced at the type level: only accredited professors may issue; only the original issuer may revoke; double-revocation is blocked by E_ALREADY_REVOKED; protocol can be paused without contract migration.

06 Walletless by Design — Across All Phases

The most deliberate architectural decision in TrustCycle: no user will ever interact with a crypto wallet — not in Phase 1, not in Phase 4. This is not a temporary tradeoff. It is the core product philosophy.

As the system evolves from custodial to fully decentralised, the technical layer matures — but the user experience remains identical: email, OTP, and a button. All cryptographic complexity is absorbed into the infrastructure.

07 Future DID Architecture — University SSO as Anchor of Trust

Phase 2 introduces true non-custodial identity — but without any wallet or seed phrase. The mechanism: University SSO login grants the right to create a DID. The DID itself is derived from a cryptographic keypair generated silently in the browser. The university verifies identity; mathematics owns it.

◈SSO does not create the DID — it grants the right to create it. The DID is derived from a local keypair. Trust is anchored in mathematics, not the institution.

When a professor moves from one university to another, their DID remains unchanged. Only a new affiliation object is added. This preserves credential history and allows verifiers to trust the identity across institutions.
